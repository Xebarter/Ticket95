# Ticket95 Organizer Analytics Dashboard Spec

## Objective

Build a modern organizer analytics dashboard at `/profile/analytics` that gives event organizers actionable insights into ticket sales, revenue, attendance, buyers, affiliates, payments, and door verification.

Scope is **organizer portfolio analytics** with an optional single-event filter — not buyer purchase history.

Primary goal: help organizers make better decisions before, during, and after events using data Ticket95 already stores.

---

# Core Design Principles

* Fast, responsive, mobile-friendly
* Clear visual hierarchy matching existing profile UI
* Interactive charts (Recharts) that answer business questions
* Every KPI should support drill-down or supporting detail nearby
* Reusable section components
* Polling (not websockets) for live-event operational metrics
* Accessible and dark/light theme compatible via existing tokens

---

# Architecture

```
/profile/analytics (client)
  → filters (event + date range)
  → GET /api/profile/analytics?eventId=&from=&to=
  → server aggregates events / orders / tickets / ticket_types /
    affiliate_commissions / verifier_sessions for the signed-in organizer
```

Auth: session required. Only events where `organizer_id = session.userId`.

---

# Dashboard Layout (Phase 1)

```
Header (context + Refresh + Export CSV)
Global Filters (event + date presets)
Overview KPIs
Live Event Strip (when event is live)
Sales & Revenue Analytics
Ticket Type Performance
Attendance Analytics
Customer Analytics
Payment & Refund Analytics
Affiliate Analytics
Verifier Analytics
Capacity / Sell-through
Rule-based Insights
Recent Orders
```

---

# Global Filters (Phase 1)

* Event: All events | single owned event
* Date range presets: Today, Yesterday, Last 7 Days, Last 30 Days, This Month, Entire Range, Custom
* All widgets react to the selected filters

Deferred filters (no schema yet): Ticket Type as global filter UI only if needed client-side; Payment Method can be chart breakdown only; Sales Channel; Gate; Verifier Device as global filters.

---

# Header (Phase 1)

* Title / description
* When a single event is selected: name, date, venue, lifecycle status, optional banner thumbnail
* Actions: Refresh, Export CSV
* Last updated timestamp

---

# Overview KPI Cards (Phase 1)

Sales

* Tickets capacity / sold / remaining
* Sell-through %
* Ticket status counts: valid, used, expired, refunded

Revenue

* Gross revenue (completed orders)
* Refunded revenue
* Net revenue (gross − refunded)

Attendance

* Checked in (used / checked_in_at)
* Remaining to check in
* Attendance rate
* Capacity usage

Orders

* Total orders (in range)
* Completed / pending / failed / refunded counts
* Average order value
* Average tickets per completed order

Deferred KPIs (no fields yet): Reserved, Complimentary, Platform fees, Taxes, Payment fees, Pending/Paid payouts, Organizer earnings beyond net − affiliate commissions.

---

# Live Event Strip (Phase 1)

Show when the selected event is currently active (event date day / live window).

Poll API ~every 10 seconds.

Include:

* Live check-ins count
* Check-ins in last few minutes (rate proxy)
* Active verifier sessions (not revoked, not expired, recently seen)
* Tickets purchased today (completed orders)
* Last successful check-in time
* Last updated / connection hint

Deferred: Current queue, failed scan log stream, websocket sync.

---

# Sales & Revenue Analytics (Phase 1)

Charts:

* Daily ticket sales / revenue over time
* Hourly sales (within selected range, especially for single day / live)
* Orders over time
* Cumulative sales
* Sell-through progress
* Order status breakdown (pie)

Chart types: Line / Bar as appropriate via Recharts.

Deferred: Forecast models, promo / channel / device revenue splits.

---

# Ticket Type Performance (Phase 1)

Table columns:

* Ticket type
* Capacity
* Sold
* Remaining
* Revenue
* Attendance (used count for that type)
* Refunds (refunded tickets of that type)
* % sold + progress bar

---

# Customer Analytics (Phase 1)

* Unique buyers
* New vs returning (returning = 2+ completed orders across selected events/range)
* Top buyers (tickets, spend, orders)
* Average spend / tickets per buyer
* Repeat purchase rate

Deferred: Age, gender, occupation, and other demographics (not collected).

---

# Payment & Refund Analytics (Phase 1)

* Revenue / order counts by `payment_provider`
* Refunded order count and amount
* Failed order count

Deferred: Chargebacks, taxes, platform fee ledger, payout status.

---

# Affiliate Analytics (Phase 1)

For orders/commissions on organizer events:

* Affiliate referral code / id
* Sales count
* Order revenue attributed
* Commission amount / status totals
* Leaderboard

---

# Attendance Analytics (Phase 1)

* Attendance rate
* Checked in vs remaining
* No-shows proxy (sold valid tickets not checked in after event, when applicable)
* Check-in timeline
* Peak entry hour

Deferred: Gate-level analytics (no gate entity), facial recognition, crowd density.

---

# Verifier Analytics (Phase 1)

From `verifier_sessions` + `tickets.checked_in_by`:

* Device name
* Session created / last seen / revoked
* Check-ins attributed to session
* Active vs expired sessions

---

# Capacity Analytics (Phase 1)

* Per-event capacity, sold, remaining, fill rate
* Portfolio rollup when “All events” selected

---

# Insights Panel (Phase 1)

Rule-based (not ML):

* Low sell-through warning
* High refund rate warning
* Strong affiliate contributor
* Strong check-in momentum during live events
* Top performing ticket type

Deferred: AI-generated narratives, forecasting recommendations.

---

# Reports & Export (Phase 1)

* CSV export of key tables (orders summary, ticket types, buyers, affiliates)
* Manual refresh

Deferred: Scheduled email reports, shareable public dashboards, saved custom reports.

---

# Security

* Session required
* Only organizer-owned events
* Sensitive financial data only for the authenticated organizer
* Prefer server-side aggregation (service role) after ownership checks

---

# Technology

* Next.js App Router API route
* supabaseAdmin for aggregation after auth
* Recharts + existing `components/ui/chart`
* Profile UI primitives (`ProfilePageHeader`, `ProfileMetric`, `ProfileSection`)
* Client hook for filters / fetch / live polling

---

# Deferred Enhancements (do not build in Phase 1)

* Multi-org / team RBAC analytics access logs
* Geographic maps / heatmaps
* Browser / OS / device analytics
* Marketing UTM / traffic source analytics
* Visitor → checkout conversion funnel
* Promo code analytics
* Taxes, platform fees, payouts
* Fraud scoring
* AI summaries / predictive demand
* Websocket realtime
* Gates as first-class entities
* Merchandise / F&B / parking / sponsors performance packs
* Benchmarking vs similar events

---

# Final Development Principles

1. Clarity before complexity.
2. Every chart answers a business question.
3. KPIs link to supporting detail on the same page.
4. Poll live metrics only when an event is live.
5. Aggregate on the server for large event portfolios.
6. Keep components modular and reusable.
7. Present insights, not only raw numbers.
8. Expand later without redesigning the page shell.
