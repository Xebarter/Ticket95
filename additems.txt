# Ticket95 Door Verifier (shipped contract)

Fast, installable, shareable ticket check-in for event doors.

## Goals

* Scan tickets in under 100ms after one-time download (local `Map` lookup).
* Support many verifier devices at once.
* Sync check-ins across devices (atomic DB update + sync poll + realtime broadcast).
* Keep working briefly offline; queue check-ins and flush when online.
* Staff never need a Ticket95 login — only a link + 6-digit code.
* Install as a Progressive Web App on phone/tablet/laptop.

## How organizers share

1. Open **Profile → Verify**.
2. For an event, click **Generate code** (or **Rotate code**).
3. Use **Share verifier** (or copy link + code). Staff can also scan the on-screen QR of the link.
4. Rotating the code revokes existing door sessions.

Share URL shape:

```
https://{host}/verify/{slug}
```

Staff enter the 6-digit access code and an optional device name (e.g. Gate A).

## How staff install

* **Android / desktop Chrome:** open the link → unlock with code → tap **Install** when prompted (Add to Home Screen).
* **iPhone:** Safari → Share → **Add to Home Screen**.
* Manifest start URL is scoped to `/verify/{slug}` so the home-screen icon opens the right event.

## Architecture (what actually ships)

```
Customer purchase → tickets row in Supabase
                         │
                         ▼
Organizer generates slug + hashed access code
                         │
                         ▼
Staff: /verify/{slug} + code → verifier_sessions JWT
                         │
                         ▼
Bootstrap GET /api/verify/tickets → IndexedDB + in-memory Map
                         │
                    Scan = Map.get (no DB)
                         │
                         ▼
Optimistic mark local → POST /api/verify/check-in (atomic)
                         │
         sync poll + broadcast channel verify:{eventId}
```

### Ticket95 field mapping

| Concept | Implementation |
|---|---|
| qrToken | `tickets.qr_code` (exact string match; paid tickets store JSON without `ticketId`) |
| used | `tickets.status === 'used'` |
| checked_in_at / checked_in_by | columns on `tickets` |
| eventSlug | `events.verify_slug` |
| access code | `events.verifier_code_hash` (plaintext shown once to organizer) |
| sessions | `verifier_sessions` |

Signed `T95.` QR envelopes, buyer-name search, NFC, face ID, etc. remain **future** — current QR payloads keep working.

### APIs

| Route | Auth | Purpose |
|---|---|---|
| `POST /api/verify/session` | access code | Issue door JWT |
| `GET /api/verify/tickets` | verifier JWT | Bootstrap lean tickets |
| `POST /api/verify/check-in` | verifier JWT | Atomic mark used |
| `GET /api/verify/sync?since=` | verifier JWT | Delta sync |
| `GET/POST /api/profile/verify/access` | organizer/admin | Link + generate/rotate code |
| `POST /api/profile/verify/scan` | organizer/admin | Organizer fallback scanner |
| `POST /api/profile/verify/mark-used` | organizer/admin | Organizer fallback mark used |

### Local cache (lean only)

```
id, qr_code, status, ticket_type_name, checked_in_at, checked_in_by, updated_at
```

Never download payment details, emails, or phones.

### Atomic check-in

```sql
UPDATE tickets
SET status = 'used', checked_in_at = now(), checked_in_by = ?
WHERE id = ? AND event_id = ? AND status = 'valid'
```

Rows affected 0 → already used / conflict.

### Offline

Successful local scans enqueue in IndexedDB (`pending`). On `online`, flush via check-in API. HTTP 409 → show conflict.

### Security

* Access codes are hashed with a server pepper (`VERIFIER_CODE_PEPPER` or `VERIFIER_JWT_SECRET`).
* Sessions expire (~24h), are individually revocable, and are bulk-revoked on code rotate.
* Old unauthenticated profile verify APIs are locked down.

### Migration

Run [`scripts/sql/023_verifier_system.sql`](scripts/sql/023_verifier_system.sql) (also appended in `run_all_migrations.sql`).

### Performance targets

* Init: seconds for typical catalogs (bulk select once).
* Lookup: Map.get, milliseconds.
* Multi-device latency: sync poll ~4s + broadcast best-effort.

## UI principles

Large buttons, high-contrast VALID / INVALID flash, counts for Loaded / Checked in / Remaining, minimal taps (auto check-in on valid scan).

## Future (not in this release)

NFC, barcodes beyond QR, face recognition, wristbands, multi-day passes, heatmaps, signed QR tokens, VIP lanes, hardware scanners.
